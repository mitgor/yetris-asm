**NEON Collision Detection Analysis (OPT-02):**

Analyze _is_piece_valid in board.s and document why NEON is not beneficial:

1. Read board.s _is_piece_valid carefully. Note the structure:
   - Outer loop: rows 0-4 of 5x5 piece grid
   - Inner loop: cols 0-4 of 5x5 piece grid
   - For each non-empty cell: compute board coordinates, do 4 bounds checks (< 0, >= 10, < 0, >= 20), then load single board byte
   - The access pattern is scattered (px+col, py+row varies per cell)

2. Explain why NEON does not help:
   - The data is 25 individual byte checks with per-cell branching on bounds
   - Board cells are accessed at computed offsets (not contiguous)
   - NEON excels at operating on contiguous vectors; this is inherently scalar (conditional, branchy, scattered)
   - The setup cost of loading piece cells into NEON registers, computing board offsets, and gathering non-contiguous board bytes exceeds the scalar loop cost
   - At most ~12 active cells per piece (T-piece has 4 cells in a 5x5 grid), so the loop body only executes 4 times for most pieces

3. Optionally: estimate instruction counts. Current scalar: ~10 instructions per active cell x 4 cells = ~40 instructions. NEON setup + gather would require at least 20+ instructions just for register preparation, plus the actual comparison, with no clear path to avoiding bounds check branching.

**Syscall Batching Analysis (OPT-04):**

Analyze the I/O path and document why syscall batching offers no benefit:

1. Trace the output path:
   - _render_frame calls multiple ncurses functions (wmove, waddch, wattr_on, etc.)
   - ncurses internally buffers all of these in a screen buffer
   - _wrefresh (called once at end of _render_frame) flushes the buffer to the terminal in a single write() syscall
   - ncurses already implements optimal batching

2. Trace the input path:
   - _wgetch with wtimeout(16) blocks for up to 16ms
   - This is a single read() syscall per frame
   - Cannot be batched (interactive input)

3. Trace the timer path:
   - _get_time_ms calls _gettimeofday (1 syscall per gravity check)
   - _mach_absolute_time (added in Plan 01) reads from commpage (no syscall on ARM64 -- it reads CNTVCT_EL0 directly)

4. Conclusion: The only syscalls per frame are: 1 write (wrefresh), 1 read (wgetch), 1 gettimeofday. These cannot be reduced further without bypassing ncurses entirely, which would be a rewrite rather than an optimization.
