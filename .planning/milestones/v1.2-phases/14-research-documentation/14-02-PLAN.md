### Section 1: Subwindow Composition (~100-120 lines)

Deep analysis of the ncurses subwindow architecture:

**1a. Problem statement:** Why not just draw to stdscr? The original C++ used LayoutGame/LayoutMainMenu classes. In assembly, we need an equivalent way to partition the 80x24 terminal into independent regions without OOP abstractions.

**1b. Window hierarchy design:** Document the complete hierarchy:
- _win_main (newwin 24x80) as root
- 8 child windows via derwin: leftmost, hold, score, middle_left, board, middle_right, rightmost, pause
- Why derwin (subwindow sharing parent memory) vs newwin (independent)
- Data storage: 12 WINDOW* pointers as .quad slots in data.s (9 game + 3 menu)

**1c. The wnoutrefresh+doupdate protocol:** This is the key technique.
- Show the full _render_frame function from render.s (lines 1803-1901)
- Explain: werase+wnoutrefresh containers first (parent before child prevents stale pixels)
- Then draw content into leaf windows (each draw function does its own wnoutrefresh)
- Board gets a second wnoutrefresh after piece/ghost overlay
- Single doupdate() at the end = one write() syscall to terminal
- Compare: if using wrefresh per window, each call would diff+write separately

**1d. Lifecycle management:** Show from layout.s:
- init_game_layout creates top-down (parent must exist before derwin child)
- destroy_game_layout deletes bottom-up (child must be deleted before parent)
- NULL pointer guards on destroy (safe to call even if windows not created)
- Window pointer zeroing after delwin (prevents use-after-free)

**1e. Coordinate systems:** Document the two mapping layers:
- Board cell (row, col) -> _win_board position (row+1, col*2+1) -- the +1 accounts for borders, *2 because each cell renders as "[]" (2 chars)
- _draw_board iterates 20 rows x 10 cols, computing wmove position per cell

**1f. Binary cost:** layout.s = 776 bytes (4.6% of __text). The 9 WINDOW* pointers add 72 bytes to __data. This is a low-cost architectural pattern.

### Section 2: Animation Double-Buffering (~100-120 lines)

Deep analysis of the background animation system:

**2a. Architecture overview:** 4 animations (fire, water, snakes, life) selected randomly at startup via arc4random_uniform(4). Dispatch table pattern: _anim_dispatch loads _anim_type, indexes into a pointer table, and branches via `br x8`.

**2b. Buffer strategy -- two different approaches:**

Fire animation uses buf1 as intensity buffer and buf2 as a static cooling map (initialized once at startup, never swapped). This is NOT true double-buffering -- it's a source+modifier pattern:
- buf1[row*80+col] = intensity (halfword, 0-100)
- buf2[row*80+col] = cooling factor (halfword, random 0-13, set once)
- Each frame: propagate heat upward in buf1, subtract cooling from buf2
- Show the propagation code from animation.s (~lines 350-430)

Water and Life animations use TRUE double-buffering with a swap flag:
- _anim_swap byte in __DATA toggles between 0 and 1
- If swap=0: read from buf1, write to buf2
- If swap=1: read from buf2, write to buf1
- Toggle swap at end of each update
- Show the swap selection code from _anim_water_update_and_draw or _anim_life_update_and_draw

Snakes uses neither -- it maintains a list of active snake entities (up to SNAKE_MAX=50) with direct screen output, no cellular automaton buffer.

**2c. Memory layout:** Two 3,840-byte buffers (80 * 24 * 2 bytes per halfword) = 7,680 bytes in __DATA. This is the largest single data allocation in the project. Halfwords (not bytes) because fire intensity values exceed 255.

**2d. Timer gating:** Each animation has its own update rate (fire: 100ms, water: 300ms, life: 200ms, snakes: 50ms move + 200ms add). Timer check uses _get_time_ms (gettimeofday wrapper). If elapsed < rate, skip to draw-only path. This prevents animations from running at frame rate (60fps would be too fast).

**2e. Draw phase:** After update (or skip), iterate all 1,920 cells (80*24). Map buffer value to char + color attribute. Use mvwaddch to write directly into _win_main. Show the fire intensity-to-color mapping code (intensity 21-40: red bold, 41-60: yellow, 61-80: yellow bold, 81+: white bold).

**2f. Binary cost:** animation.s = 3,384 bytes (20.3% of __text) -- the second-largest file. The animation buffers add ~7,680 bytes to __DATA. Background animations are the single most expensive v1.1 feature in binary size terms, contributing 51.3% of the v1.0->v1.2 __text growth.

### Section 3: Modern Guideline Scoring Pipeline (~80-100 lines)

Deep analysis of the scoring engine in _lock_piece (board.s):

**3a. Scoring model:** Modern Tetris guideline scoring as implemented by NES-era fan communities. Not the original NES scoring (which was simpler) but the modern guideline with T-spins, combos, back-to-back, and perfect clears.

**3b. T-spin detection algorithm:**
- Precondition: piece_type == 6 (T-piece) AND _last_was_rotation == 1
- Three-corner rule: check 4 corners around pivot cell (piece_y+2, piece_x+2)
- Each corner: bounds check (out-of-bounds counts as occupied) then board cell check
- If 3+ corners occupied: is_tspin = 1
- Show the corner-check code from board.s (lines ~276-380)
- Why the pivot is at grid(2,2): SRS uses 5x5 grid with piece center at (2,2)

**3c. Score calculation pipeline (sequential, no branches skipped):**
1. Base score: _score_table[lines-1] for normal, _tspin_score_table[lines] for T-spin
   - Normal: 100/300/500/800 for 1/2/3/4 lines
   - T-spin: 400/800/1200/1600 for 0/1/2/3 lines
2. Multiply by _level: `mul w10, w10, w11`
3. Back-to-back check: "difficult" = Tetris (4 lines) OR T-spin with lines > 0
   - If b2b_active AND is_difficult: `add w10, w10, w10, lsr #1` (integer 1.5x without FP)
   - Show this instruction and explain: lsr #1 = divide by 2, add original = multiply by 1.5
4. Add to _score
5. Combo: increment _combo_count, add 50 * combo_count * level
6. Perfect clear: if all 200 board cells are 0 after line clear, add bonus

**3d. State management:** 5 __DATA variables track scoring state: _score (uint32), _combo_count (uint32), _b2b_active (byte), _is_tspin (byte), _last_was_rotation (byte). These are reset in _reset_board.

**3e. The integer 1.5x pattern:** `add w10, w10, w10, lsr #1` is the most elegant ARM64 idiom in the scoring engine. It avoids floating-point entirely: `result = value + (value >> 1) = value * 1.5`. This works because ARM64 shifted-register operands execute in a single cycle. On x86-64, this would be 2 instructions (shr + add) or a lea with complex addressing.

**3f. Binary cost:** The scoring engine is embedded in _lock_piece in board.s. board.s total = 2,400 bytes (14.4%). The scoring section itself is approximately ~400 bytes within _lock_piece.

### Section 4: Hi-Score File I/O via Darwin Syscalls (~80-100 lines)

Deep analysis of raw syscall file I/O:

**4a. Why raw syscalls?** The C library provides open/read/write/close wrappers, but they add function call overhead (PLT stub -> GOT -> dyld resolution -> actual libc function). Raw syscalls via `svc #0x80` bypass this entirely. For a 4-byte file read/write, the libc overhead per call (~20 instructions of PLT+GOT resolution) exceeds the actual work.

Actually, clarify the real reason: it's a research project demonstrating ARM64 techniques. The syscall approach shows how Darwin ARM64 syscalls work at the hardware level, which is the research value. The performance benefit is irrelevant (file I/O happens twice: once at startup, once at game over).

**4b. Darwin ARM64 syscall convention:**
- x16 = syscall number (#5=open, #3=read, #4=write, #6=close)
- x0-x5 = arguments (same as function call ABI)
- svc #0x80 triggers supervisor call
- Return: x0 = result (fd, bytes read/written, or errno)
- Error: carry flag set (CS condition). This differs from Linux (which returns negative errno in x0).
- Show the complete _load_hiscore function from hiscore.s

**4c. Path construction on the stack:**
- Allocate 256-byte buffer via `sub sp, sp, #256`
- getenv("HOME") returns pointer to HOME string
- Byte-by-byte copy HOME to stack buffer (ldrb+strb+cbnz loop)
- Append "/.yetris-hiscore" suffix
- This avoids malloc entirely -- pure stack allocation

**4d. File format design:**
- Raw uint32 little-endian, 4 bytes, no header, no text
- Why not human-readable text? Text requires integer-to-string conversion (expensive in assembly -- division loop) and string-to-integer parsing (also expensive). Binary format: one ldr/str instruction.
- Why uint32 not uint64? Maximum reasonable Tetris score fits in 32 bits (~4 billion). 64-bit would waste file space and require ldp/stp instead of ldr/str.

**4e. Error handling pattern:**
- `b.cs Lbail` after every syscall -- carry flag set means error
- Bail means: clean up stack (free buffers), restore callee-saved registers, return
- _hiscore remains at its current value (0 on first run, previous value if save fails)
- This is "fail-silent" design: the game works fine without persistence

**4f. Stack frame discipline:**
- _load_hiscore and _save_hiscore use identical frame layout: callee-saved pair (x19, x20), frame record (x29, x30), path buffer (256 bytes), optional read/write buffer (16 bytes)
- x19 = path buffer base (survives bl _getenv), x20 = file descriptor (survives syscalls)
- The 16-byte read/write buffer is allocated AFTER open succeeds and freed BEFORE close -- minimal stack usage

**4g. Binary cost:** 352 bytes total (2.1% of __text) for both load and save functions. This is the smallest feature module, demonstrating that file I/O in assembly is surprisingly compact when the format is binary.

### Section 5: Cross-Cutting Patterns (~40-50 lines)

Patterns that appear across all 4 techniques:

**5a. adrp+add / adrp+ldr page-relative addressing:**
Every technique uses this pattern to access data.s globals. adrp loads the page base (4KB-aligned), add/ldr adds the page offset. This is ARM64's answer to x86-64's RIP-relative addressing, but split across two instructions due to fixed 32-bit instruction encoding.

**5b. Callee-saved register discipline:**
All functions use x19-x28 for values that must survive `bl` calls. The pattern: stp pairs in prologue, ldp pairs in epilogue, exact frame size. No alloca, no variable-length frames.

**5c. Timer-gated updates:**
Both the gravity system (main.s) and animations (animation.s) use _get_time_ms + elapsed comparison to gate updates. This pattern decouples update rate from frame rate.

**5d. Table-driven design:**
Score tables, animation dispatch tables, piece data tables -- the project consistently replaces conditional logic with table lookups. One adrp+add+ldr replaces a chain of cmp+b.eq branches.

### Conclusion (~10-15 lines)

Summarize: the v1.1 techniques demonstrate that assembly is not just about low-level optimization -- it's about implementing real software engineering patterns (window hierarchies, double-buffering, scoring pipelines, file persistence) at the instruction level. The techniques added 6,600 bytes to a 10,096-byte codebase, a 65% increase that delivered 4 major features. The code remains readable with consistent commenting and register discipline.
