  <name>Task 1: Identify and implement size optimizations</name>
  <files>asm/data.s, asm/render.s</files>
  <action>
**Before any changes, record baseline:**
```
ls -la asm/bin/yetris-asm
size -m asm/bin/yetris-asm
```

**Investigation phase -- check each optimization opportunity:**

1. **Alignment padding audit (highest likelihood):**
   ```
   grep -n ".p2align" asm/*.s
   ```
   The Mach-O linker pads each segment to page boundaries (16KB). Check if any `.p2align` directives inside sections create unnecessary internal padding. Specifically:
   - `.p2align 4` (16-byte align) in data sections wastes bytes if data is naturally smaller-aligned
   - `.p2align 2` (4-byte align) before functions is standard and should be kept
   - Check if reducing `__const` alignment from `.p2align 4` to `.p2align 2` saves internal padding

2. **Duplicate/similar string detection:**
   ```
   grep -n '.asciz' asm/data.s | sort -t'"' -k2
   ```
   Look for duplicate string content or strings that share suffixes (e.g., "Level" appearing in multiple strings). String deduplication could save bytes if duplicates exist.

3. **Data section consolidation:**
   Check if any `__DATA,__data` items could be moved to `__TEXT,__const` (read-only). Items that are initialized once and never modified are candidates. However, ARM64 ABI requires mutable data in __DATA, so only truly immutable data qualifies.

4. **Unused GOT entry check:**
   ```
   nm asm/bin/yetris-asm | grep -c " U "
   otool -l asm/bin/yetris-asm | grep -A2 "__got"
   ```
   Each GOT entry is 8 bytes. Verify all imported symbols are actually called. (After Phase 12 cleanup this is likely clean, but verify.)

**Implementation phase -- apply at least one optimization:**

For each optimization that yields measurable savings:
- Make the change
- Run `make asm-clean && make asm`
- Record new binary size: `ls -la asm/bin/yetris-asm && size -m asm/bin/yetris-asm`
- Document the before/after delta

If NO optimization yields measurable savings at the linked binary level (due to page-boundary rounding), document this as a finding: individual section savings may be absorbed by page padding. In this case, document the per-section savings even if file size doesn't change.

**IMPORTANT:** The game must still build and run correctly after any changes. If an optimization breaks the build, revert it. Use `make asm-run` for a quick smoke test (launch game, verify main menu appears, start a game, drop a piece).
