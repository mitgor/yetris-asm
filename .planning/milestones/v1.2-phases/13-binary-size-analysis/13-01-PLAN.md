  <action>
Run the following measurement commands against the CURRENT binary (asm/bin/yetris-asm) and create research/binary-size-analysis.md with the results:

1. **Overall binary file size:**
   ```
   ls -la asm/bin/yetris-asm
   ```
   Record unstripped file size in bytes.

2. **Mach-O section breakdown (SIZE-01):**
   ```
   size -m asm/bin/yetris-asm
   ```
   Document every section: __text, __stubs, __const (in __TEXT), __got, __const (in __DATA_CONST), __data (in __DATA), and LINKEDIT. Show both individual section sizes and segment totals.

   Also run:
   ```
   otool -l asm/bin/yetris-asm
   ```
   Extract exact byte counts from the `size` field of each Section entry for precision.

3. **Per-file __text contribution (SIZE-02):**
   Run for each .o file:
   ```
   size -m asm/*.o
   ```
   Create a table showing each source file's __text (code) bytes, __const bytes, and __data bytes. Sum should match the linked binary's section sizes (note: linker may add padding/alignment, document any discrepancy).

   Sort the per-file table by __text size descending (render.s is expected to be largest).

4. **Per-file percentage breakdown:**
   Calculate what percentage of total __text each file contributes. This shows where code "mass" lives.

5. **Symbol count analysis:**
   ```
   nm asm/bin/yetris-asm | wc -l
   nm asm/bin/yetris-asm | grep -c " T "   # text symbols
   nm asm/bin/yetris-asm | grep -c " D "   # data symbols
   nm asm/bin/yetris-asm | grep -c " U "   # undefined (imported)
   ```

6. **Stripped binary comparison:**
   ```
   ls -la asm/bin/yetris-asm-stripped
   ```
   If the stripped binary exists, note its size. If not, create it:
   ```
   cp asm/bin/yetris-asm /tmp/yetris-asm-stripped && strip /tmp/yetris-asm-stripped && ls -la /tmp/yetris-asm-stripped
   ```

7. **Growth from v1.0 to post-v1.1:**
   Include a "Binary Growth" section showing the v1.0 baseline from the existing writeup (55,672 bytes, __text=10,096) vs current measurements, with delta and percentage growth. List the v1.1 features responsible for the growth (animation system ~3,384 bytes of __text, hi-score persistence ~352 bytes, subwindow rendering, etc. -- based on per-file measurements).

8. **Document structure:**
   The file should have these sections:
   - Title: "Binary Size Analysis: yetris-asm (post-v1.1/v1.2)"
   - Date of measurement
   - Section Breakdown (table from step 2)
   - Per-File Contributions (table from step 3, sorted by code size)
   - Symbol Analysis (from step 5)
   - Binary Growth (v1.0 vs current, from step 7)
   - Stripped Binary (from step 6)
   - Notes/methodology

All numbers MUST come from actual measurements, not from the stale writeup. Run the commands and use the output directly.
