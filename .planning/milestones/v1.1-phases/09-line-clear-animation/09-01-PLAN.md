**board.s changes -- new `_mark_lines` function:**
Create `_mark_lines` as a new global function. It replaces the `_clear_lines` call in `_lock_piece`.

Behavior:
- Scan board bottom-to-top for full rows using the EXACT same NEON uminv logic as `_clear_lines` (ld1 16b, orr with _neon_row_mask, uminv, umov, cbz)
- For each full row: overwrite ALL 10 cells with value 9 (flash marker). Do NOT shift rows.
- Count lines marked. Update `_lines_cleared`, `_level`, and line-type stats (`_stats_singles` etc.) -- same stat update logic as `_clear_lines` Lclear_done section.
- After marking: if lines_marked > 0, set `_line_clear_state = 1` and record current time via `bl _get_time_ms` into `_line_clear_timer`.
- Return lines_marked count in w0.

Register allocation: Same as `_clear_lines` -- x19=board base, w20=row counter, w21=lines count, w22=col/byte counter. Stack: 64 bytes.

Key difference from `_clear_lines`: The marking loop is simpler. When a full row is found, do NOT shift. Just overwrite 10 cells with `mov w8, #9 / strb w8, [x19, x10]` in a 10-iteration fill loop. Then continue scanning upward (decrement row counter, unlike `_clear_lines` which re-checks same row after shifting).

**board.s changes -- new `_clear_marked_lines` function:**
Create `_clear_marked_lines` as a new global function. Called from game loop after 200ms delay.

Behavior:
- Scan board bottom-to-top (row 19 to 0)
- For each row: check if cell[0] == 9 (row is marked)
- If marked: shift all rows above down by one (EXACT same shift logic as `_clear_lines` Lclear_shift_loop -- copy row j-1 to row j, fill top with zeros). After shifting, re-check same row index (don't decrement).
- If not marked: decrement row counter and continue.
- No stats, no scoring, no return value. Just collapse.

Register allocation: Same as `_clear_lines` shift section. Stack: 64 bytes.

**board.s changes -- modify `_lock_piece`:**

Do NOT delete `_clear_lines` -- keep it as dead code for reference until the new functions are proven.

**main.s changes -- animation state machine in game loop:**
Insert a line-clear animation check between the pause check and gravity check. Specifically, after `Lskip_gravity` is the target for the pause branch (line 198-199), but the animation check goes BEFORE gravity, after the pause test.

The current flow at lines 197-237 is:
```

```

Change to insert a line-clear state check between the pause branch and gravity. Right after the pause branch (after line 199 `b.ne Lskip_gravity`), add:

```asm
    // Check line clear animation state
    adrp    x8, _line_clear_state@PAGE
    ldrb    w9, [x8, _line_clear_state@PAGEOFF]
    cbnz    w9, Lflash_active

    // ... existing gravity code unchanged ...

Lflash_active:
    // Flash animation is active -- check if 200ms delay has expired
    bl      _get_time_ms                // x0 = current time ms
    mov     x19, x0                     // save current time
    adrp    x8, _line_clear_timer@PAGE
    ldr     x9, [x8, _line_clear_timer@PAGEOFF]
    sub     x10, x19, x9               // elapsed = current - start
    cmp     x10, #200
    b.lt    Lskip_gravity               // still flashing, skip gravity, go to render

    // Flash expired: clear marked lines, spawn next piece
    bl      _clear_marked_lines
    bl      _spawn_piece

    // Reset line clear state
    adrp    x8, _line_clear_state@PAGE
    strb    wzr, [x8, _line_clear_state@PAGEOFF]

    // Reset gravity timer (prevent instant drop after flash, same as pause resume)
    bl      _get_time_ms
    adrp    x8, _last_drop_time@PAGE
    str     x0, [x8, _last_drop_time@PAGEOFF]

    // Sync game_over from _spawn_piece (new piece may collide)
    adrp    x8, _game_over@PAGE
    ldrb    w9, [x8, _game_over@PAGEOFF]
    bic     x28, x28, #1               // clear bit 0
    orr     x28, x28, x9               // bit 0 = game_over

    b       Lskip_gravity               // continue to render
```

Place the `Lflash_active` block before `Lskip_gravity` (between the gravity code and the Lskip_gravity label). The key: during flash, we skip ALL input processing AND gravity. Since the flash check is after input handling but before gravity, and we jump directly to Lskip_gravity (which leads to render), input was already processed in the current frame -- but since there's no active piece during flash, movement/rotation keys are no-ops (they call _try_move/_try_rotate which check piece validity against the board). The only concern was pause/quit, which are handled before the flash check via the existing x28 pause/game_over sync.

IMPORTANT: The `Lflash_active` code block uses x19 (current time). x19 is callee-saved and is used by the gravity code below for the same purpose. Since `Lflash_active` branches to `Lskip_gravity` (skipping gravity), there is no conflict.
