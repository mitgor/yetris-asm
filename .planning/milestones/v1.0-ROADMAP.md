# Roadmap: yetris-asm

## Overview

This roadmap takes the yetris ARM64 assembly rewrite from zero to a fully playable, feature-complete Tetris clone with documented optimization research. The progression is: validate the Darwin toolchain and ABI conventions (Phase 1), build a minimal playable Tetris (Phase 2), add the remaining gameplay features and establish the binary size baseline (Phase 3), add menus and game mode configuration (Phase 4), then pursue the assembly optimization research that motivates the project (Phase 5). Each phase produces a runnable binary that does more than the last.

## Phase Details

### Phase 1: Foundation and Darwin ABI Scaffold
**Goal**: A working AArch64 assembly project that builds, links, code-signs, and runs on macOS Apple Silicon, with ncurses terminal I/O proven correct
**Depends on**: Nothing (first phase)
**Requirements**: FOUN-01, FOUN-02, FOUN-03, FOUN-04
**Success Criteria** (what must be TRUE):
  1. Running `make asm` (or equivalent target) in the project root assembles, links, code-signs, and produces a runnable Mach-O binary in asm/ without errors
  2. The binary initializes ncurses, displays text on screen, waits for a keypress, and exits cleanly (no terminal corruption, cursor restored)
  3. The Makefile builds assembly targets in asm/ without breaking the existing C++ build in src/ -- both `make` and `make asm` succeed independently
  4. The assembly source demonstrates correct Darwin ABI patterns: x16 syscalls, 16-byte stack alignment, x18 avoidance, underscore-prefixed symbols, adrp+add addressing

### Phase 2: Core Playable Game
**Goal**: A playable Tetris game in assembly -- pieces fall, the player moves and rotates them, lines clear, score increments, and the game ends when the board fills
**Depends on**: Phase 1
**Requirements**: MECH-01, MECH-02, MECH-03, MECH-04, MECH-05, MECH-06, MECH-07, MECH-08, MECH-12, MECH-14, MECH-15, REND-01, REND-02, REND-03, REND-05
**Success Criteria** (what must be TRUE):
  1. All 7 tetrominoes spawn, fall under gravity, and rotate using SRS wall kicks -- pieces visibly shift when kicked off walls
  2. The player can move pieces left/right, soft drop, and hard drop (instant placement) using keyboard input
  3. Completed rows flash or disappear and the rows above collapse downward; the score panel updates correctly (100/300/500/800 for 1/2/3/4 lines)
  4. The game ends with a visible indication when a new piece spawns into an occupied cell; restarting requires re-running the binary
  5. Gravity speed visibly increases as the level advances (level 1 pieces fall noticeably slower than level 10+ pieces)

### Phase 3: Gameplay Feature Completeness
**Goal**: The assembly Tetris matches the yetris C++ feature set for in-game mechanics, and the first binary size comparison against the C++ baseline is recorded
**Depends on**: Phase 2
**Requirements**: MECH-09, MECH-10, MECH-11, MECH-13, REND-04, MEAS-01
**Success Criteria** (what must be TRUE):
  1. A translucent or distinct ghost piece shows exactly where the current piece will land, updating in real-time as the player moves
  2. The player can hold the current piece (swapping it into a hold slot) and it displays in a side panel; holding again before the next piece locks is rejected
  3. The next 1+ pieces are visible in a preview panel, and a statistics panel shows piece counts and line clear counts (singles/doubles/triples/tetris)
  4. The player can pause the game (timer suspends, board obscured or frozen) and resume without losing state
  5. Binary size of the assembly version is measured and recorded alongside the C++ yetris binary size for the same feature set

### Phase 4: Menus and Game Modes
**Goal**: The game has a main menu, help screen, and configurable game modes -- it launches into a menu, not directly into gameplay
**Depends on**: Phase 3
**Requirements**: UI-01, UI-02, UI-03, UI-04
**Success Criteria** (what must be TRUE):
  1. The binary launches to a main menu where the player can choose to start a game, view help, or quit
  2. A help screen displays all keybindings and controls; the player can return to the main menu from help
  3. The player can select a starting level (1-22) before starting a game, and the game begins at that level's gravity speed
  4. Game mode toggles (initial noise, invisible mode, ghost piece on/off, hold on/off) are configurable from the menu and take effect in-game

### Phase 5: Optimization Research and Documentation
**Goal**: Assembly-specific optimization techniques are explored, measured with quantitative before/after results, and documented in a research writeup comparing the assembly version to the C++ baseline
**Depends on**: Phase 4
**Requirements**: OPT-01, OPT-02, OPT-03, OPT-04, OPT-05, OPT-06, MEAS-02, MEAS-03, MEAS-04, MEAS-05
**Success Criteria** (what must be TRUE):
  1. At least one optimization technique (register packing, NEON collision, NEON line detection, or syscall batching) is implemented with measured before/after performance numbers
  2. Frame timing data is collected using mach_absolute_time or clock_gettime, showing per-frame cost in microseconds for the assembly version
  3. The binary has been analyzed with size/otool/nm and optimized with dead_strip and symbol stripping; final binary size is compared to the C++ baseline in a table
  4. A research writeup document exists that describes each technique attempted, the quantitative results, what worked, what did not, and which optimizations are ARM64-specific vs generic
  5. CPU profiling with Instruments has identified the actual hotspots in the assembly game loop, and the writeup discusses whether those hotspots match expectations
**Plans**: 3 plans

Plans:

- [x] 05-02-PLAN.md -- NEON line detection + register state packing + before/after measurements
- [x] 05-03-PLAN.md -- NEON collision analysis + syscall batching analysis + comprehensive research writeup
